# Decisiones y desviaciones

Este documento registra decisiones tecnicas relevantes y desviaciones necesarias respecto a la metodologia comun.

## Estado actual del desarrollo

- Se crea un monorepo unico.
- Se define `benchmark_core` como paquete comun para contratos, metricas y persistencia.
- `ARCH_01_SINGLE_REACT` queda implementada en LangGraph y CrewAI con un agente/nodo decisor unico.
- `ARCH_02_SEQUENTIAL_PIPELINE` queda implementada en LangGraph y CrewAI como pipeline secuencial de cuatro fases: planner, retriever, analyst y writer.
- `ARCH_03_ROUTER_SPECIALISTS` queda implementada en los cinco frameworks con router explicito, tres especialistas logicos y sintesis final.
- `ARCH_04_SUPERVISOR_WORKERS` queda implementada en los cinco frameworks con supervisor centralizado, plan inicial, decisiones iterativas, revision de salidas, workers condicionales, soporte de revisiones y finalizacion acotada.
- `ARCH_05_HANDOFF_SWARM` queda implementada en los cinco frameworks como arquitectura descentralizada de especialistas con handoffs secuenciales, sin supervisor central ni manager jerarquico.
- `psutil` queda como dependencia opcional para monitorizacion de recursos; el codigo no falla si no esta instalado.
- En `ARCH_01_SINGLE_REACT` no se introducen herramientas externas reales en la primera version. Los documentos se pasan como contexto en el prompt comun mediante `benchmark_core.render_single_react_prompt`.
- Se acepta esta variante minima porque valida schemas, runner comun, instrumentacion, persistencia JSON y comparabilidad basica antes de anadir `document_search` u otras herramientas.
- Se incorpora `OpenAIInstrumentedLLM` para pruebas con API real cuando `ExperimentConfig.model_provider == "openai"`, manteniendo el modo local determinista como baseline reproducible.
- En `ARCH_02_SEQUENTIAL_PIPELINE`, `Retriever` se implementa inicialmente como fase LLM instrumentada y no como herramienta determinista. Esto fuerza cuatro llamadas LLM por ejecucion y facilita medir el coste completo de un pipeline secuencial.
- En `ARCH_03_ROUTER_SPECIALISTS`, el router decide una sola vez que especialistas ejecutar y los omitidos quedan registrados. El orden de ejecucion sigue siendo canonico para preservar comparabilidad.
- Esta arquitectura no representa supervision real: no hay revision iterativa, feedback ni correcciones. Una futura arquitectura supervisora debe implementarse por separado.
- En CrewAI, `ARCH_03_ROUTER_SPECIALISTS` usa `Process.sequential` con un agente router explicito, en lugar de `Process.hierarchical`, para evitar comportamiento de manager implicito dificil de comparar en el estado actual del desarrollo.
- La seleccion del router se normaliza mediante un formato parseable (`SELECTED_SPECIALISTS=...`, `SKIPPED_SPECIALISTS=...`). Si un modelo real no respeta el formato, se aplica una politica conservadora para evitar ejecuciones vacias.
- Microsoft Agent Framework usa un `BaseAgent` custom que invoca el SDK oficial de OpenAI Responses. Se eligio esta ruta despues de comprobar que `OpenAIChatClient` y `OpenAIChatCompletionClient` fallaban en este entorno con un error interno de `ContextVar` de telemetria.
- Se corrige la numeracion metodologica: la supervision real se separa de `ARCH_03_ROUTER_SPECIALISTS` y pasa a `ARCH_04_SUPERVISOR_WORKERS`. `ARCH_03` queda documentada como router sin revision iterativa; `ARCH_04` concentra la sobrecarga de coordinacion, revision y posibles iteraciones.
- `ARCH_04_SUPERVISOR_WORKERS` usa `max_supervisor_iterations = 3` por defecto sin anadir campos al schema comun. Si se necesita ajustar el limite por experimento, se usa `config.metadata["max_supervisor_iterations"]` o `MAX_SUPERVISOR_ITERATIONS`.
- Se amplia el wrapper LLM determinista con prompts y respuestas parseables para `ARCH_04_SUPERVISOR_WORKERS`. Esta ampliacion no cambia la metodologia; permite ejecutar tests locales sin API externa y mantener medicion comparable de llamadas, tokens y pasos.
- Se integra `benchmark_core.result_writer.save_result_json` en los decorators de framework para cumplir el patron comun `results/raw/{framework}/{architecture}/{run_id}.json` sin duplicar guardado en cada arquitectura.
- En CrewAI, `ARCH_04_SUPERVISOR_WORKERS` usa `Process.hierarchical` con `manager_agent` nativo porque es la primitiva especifica del framework para manager/supervisor con delegacion y validacion de resultados. Para preservar tests locales sin API externa, el adaptador LLM determinista de CrewAI reconoce el formato nativo `delegate_work_to_coworker` y registra las llamadas por rol.
- Revision de primitivas nativas para `ARCH_04`: LangGraph mantiene `StateGraph` con conditional edges porque `langgraph_supervisor` no esta instalado y `StateGraph` es la API nativa disponible para control ciclico; Microsoft Agent Framework mantiene workflow/orquestacion centralizada con su paquete instalado; LlamaIndex mantiene `Workflow`/`AgentWorkflow` como abstraccion nativa disponible; Pydantic AI mantiene modelos Pydantic y `pydantic-graph`, que es la primitiva nativa para grafo tipado.
- `ARCH_05_HANDOFF_SWARM` comparte solo semantica pura (`HandoffDecision`, prompts, parsing, limites y respuestas deterministas locales). La orquestacion, seleccion del siguiente agente, estado del swarm y limites se implementan dentro de cada framework para evitar esconder el controlador completo en utils.
- Auditoria de `benchmark_core/handoff_swarm.py`: el modulo contiene solo semantica comun, prompts, schemas parseables, limites puros, normalizacion/fallback y respuestas deterministas para tests. No contiene ejecucion de agentes, bucle de handoffs, mutacion de estado operativo, routing efectivo ni maquina de estados compartida.
- LangGraph `ARCH_05_HANDOFF_SWARM` ya usaba `StateGraph`, nodos especialistas, estado tipado, `conditional_edges`, `END` y transiciones directas entre especialistas; no se modifica.
- Microsoft Agent Framework instalado para `ARCH_05`: `agent-framework-core==1.9.0`, `agent-framework-openai==1.8.2` y `agent-framework-orchestrations==1.0.0`. Se anade `agent-framework-orchestrations` al extra `microsoft_agent_framework` y a `all-frameworks` porque `HandoffBuilder` es la primitiva nativa que representa handoffs descentralizados. La implementacion de Microsoft `ARCH_05_HANDOFF_SWARM` usa `HandoffBuilder`, `with_start_agent`, `add_handoff`, herramientas nativas `handoff_to_*` y eventos `handoff_sent`. Para conservar la instrumentacion comun, el cliente benchmark transforma cada `HandoffDecision` parseable en una llamada de herramienta nativa cuando `action=handoff`; la transferencia efectiva la realiza el workflow nativo, no un bucle Python externo.
- En CrewAI `1.14.6`, `ARCH_05_HANDOFF_SWARM` se migra de un bucle Python con `Agent`/`Task` dinamicas a `CrewAI Flow` con `@start`, `@listen` y `@router`. El Flow materializa el estado y las transiciones; cada especialista sigue ejecutandose como `Agent`/`Task` CrewAI y la decision semantica del siguiente destino procede del output del agente activo. No se usa `Process.hierarchical` ni `manager_agent`, reservados para `ARCH_04_SUPERVISOR_WORKERS`.
- En LlamaIndex `llama-index-core==0.14.22`, `AgentWorkflow` y `FunctionAgent.can_handoff_to` existen. Se evaluo su uso para `ARCH_05`, pero el modo local reproducible del benchmark usa `DeterministicLlamaIndexAgent`, que no es `BaseWorkflowAgent` ni LLM function-calling de LlamaIndex. Usar `AgentWorkflow` real con handoff tool en local ocultaria o saltaria el wrapper comun de llamadas/tokens; usarlo solo en OpenAI anadiria una capacidad distinta al framework frente al baseline local. Por eso se conserva `FunctionAgent`/adaptador y workflow explicito instrumentado, documentando que no se afirma como handoff nativo de `AgentWorkflow`.
- En Pydantic AI + `pydantic-graph==1.107.0`, `ARCH_05_HANDOFF_SWARM` usa la API moderna `GraphBuilder`: cuatro `Step` especialistas desembocan en un `Decision` compartido que dirige el siguiente handoff o el paso final. La migracion elimina el aviso de deprecacion de `Graph` sin cambiar prompts, numero de llamadas, limites, trazas ni semantica secuencial.
- No se usan checkpoints, memoria persistente, handoffs descentralizados, fan-out/fan-in paralelo ni observabilidad externa en `ARCH_04_SUPERVISOR_WORKERS`.
- No se usan checkpoints, memoria persistente, fan-out/fan-in paralelo ni observabilidad externa en `ARCH_05_HANDOFF_SWARM`.
- `ARCH_06_PARALLEL_FANOUT_FANIN` queda implementada en los cinco frameworks con cuatro perspectivas independientes sobre el mismo `ExperimentInput` y un unico agregador posterior.
- La semantica comun de ARCH_06 (prompts, normalizacion de salidas parciales, baseline determinista y resumen de metricas por rama) reside en `benchmark_core.parallel_fanout_fanin`; la orquestacion permanece dentro de cada framework y no se oculta en una utilidad compartida.
- No se modifica ningun schema comun. Se usa `ExperimentMetrics.metadata`, ya existente, para añadir `latency_total_ms` y el resumen `parallel_execution`; `structured_output` conserva las salidas parciales, ramas completadas/fallidas y el indicador de paralelismo. El cambio es aditivo y compatible hacia atras.
- LangGraph ARCH_06 usa `StateGraph`, cuatro aristas desde `START` y una arista de origen multiple hacia `aggregator`. Las ramas escriben claves de estado aisladas para evitar colisiones concurrentes.
- Microsoft Agent Framework ARCH_06 usa `WorkflowBuilder.add_fan_out_edges` y `add_fan_in_edges`, que materializan `FanOutEdgeGroup` y `FanInEdgeGroup`. Los cuatro `Executor` de rama se ejecutan concurrentemente y el agregador recibe una lista sincronizada.
- CrewAI ARCH_06 usa cuatro `Task(async_execution=True, context=[])` y una tarea sincronica `aggregator` con `context` explicito a las cuatro tareas. `Crew._execute_tasks` lanza futuros para las tareas asincronas y los resuelve antes de la siguiente tarea sincronica. Se desactivan delegacion, memoria, cache, planning y proceso jerarquico.
- La adaptacion CrewAI LLM asigna ids de llamada con un lock corto y ejecuta la llamada fuera del lock, evitando ids duplicados sin serializar las ramas paralelas. Tambien se propaga `timeout_seconds` mediante `Agent.max_execution_time`. Ambos cambios son compatibles con ARCH_01 a ARCH_05.
- LlamaIndex ARCH_06 usa `Workflow`, cuatro eventos especializados, cuatro pasos concurrentes y `Context.collect_events` como fan-in. Cada paso invoca un `FunctionAgent` real para OpenAI o el adaptador determinista instrumentado para el baseline local.
- Pydantic AI ARCH_06 usa la API moderna `pydantic_graph.GraphBuilder`, un broadcast fork, cuatro `Step`, un `Join` con reducer y un `ParallelGraphOutput` validado. ARCH_05 y ARCH_06 quedan libres de la API `Graph` legacy y de bucles controladores en Python.
- En las versiones instaladas y auditadas no se requiere fallback secuencial para ARCH_06. `parallelism_used=true` y `fallback_sequential=false` en los cinco frameworks. Si una version futura elimina una primitiva, el resultado debera declarar y documentar el fallback antes de compararse.
- Para Microsoft, LlamaIndex y Pydantic AI, las llamadas sincronicas instrumentadas se encapsulan con `asyncio.to_thread` dentro de executors/pasos nativos. Esto evita bloquear el event loop sin sustituir los agentes o workflows del framework y no añade llamadas LLM.
- El smoke OpenAI de ARCH_06 valida cinco llamadas, solapamiento temporal de las cuatro ramas, fan-in posterior, trazas y metricas por componente en los cinco frameworks. Los cinco adaptadores normalizan los contadores reales de OpenAI (`input_tokens`/`output_tokens` en Responses, `prompt_tokens`/`completion_tokens` en Chat Completions y `RunUsage` en Pydantic AI) bajo `token_counting_method=openai_usage`. Si el SDK no expone `usage`, la llamada falla explicitamente: no se sustituye silenciosamente por un proxy. El conteo aproximado por palabras queda reservado a ejecuciones locales deterministas, que deben analizarse separadas de las ejecuciones OpenAI.
- `ARCH_07_MAP_REDUCE_AGENTIC` queda implementada en los cinco frameworks como Map-Reduce por volumen documental: particionado determinista, mappers equivalentes por batch y un unico reducer global. No se introducen roles especializados, routing, handoffs, debate ni supervision iterativa.
- El batch size se configura mediante `config.metadata["map_reduce_batch_size"]`, con alias `batch_size` y valor por defecto `3`. Los batches son contiguos, estables y preservan el orden de `input_data.documents`; no se modifica `ExperimentConfig` ni ningun schema comun.
- `benchmark_core.map_reduce_agentic` contiene solo semantica compartida: particionado puro, prompts, modelos de salida parcial/reducida, baseline determinista, pasos y resumen de metricas. Cada framework conserva su propia orquestacion.
- Las metricas aditivas de ARCH_07 se guardan en `structured_output.map_reduce_execution` y `ExperimentMetrics.metadata.map_reduce_execution`: documentos, batches, mappers, latencias, llamadas, tokens, errores, throughput y tipo de paralelismo. El reducer recibe solo salidas parciales y la lista de IDs originales, nunca el contenido documental completo.
- LangGraph ARCH_07 usa `StateGraph` y `Send` dinamico; Microsoft usa `WorkflowBuilder.add_fan_out_edges/add_fan_in_edges`; CrewAI usa tareas mapper `async_execution=True` y reducer con `context`; LlamaIndex usa eventos dinamicos, workers y `collect_events`; Pydantic AI usa `GraphBuilder.map`, `Step` y `Join` tipados.
- No existe fallback secuencial con las versiones instaladas. `parallelism_used` solo es verdadero cuando la ejecucion contiene al menos dos batches; con un unico batch no hay concurrencia observable y `fallback_sequential` sigue siendo falso.
- CrewAI necesita materializar las tareas despues del particionado y antes de `kickoff`, porque el grafo de tareas no admite expansion durante la ejecucion. Los mappers siguen siendo tareas nativas equivalentes y concurrentes; esta diferencia de construccion no altera llamadas ni prompts.
- El smoke OpenAI sintetico de ARCH_07 usa siete documentos inventados, `batch_size=3`, tres mappers y un reducer: valida 20 llamadas totales, cobertura documental exacta, solapamiento temporal real, fan-in posterior, metricas y persistencia raw en los cinco frameworks. Es una prueba de integracion, no un benchmark de calidad; los resultados publicables se obtendran posteriormente con un dataset externo versionado.
- Los `FunctionAgent` de LlamaIndex se configuran con `streaming=False` en ejecuciones OpenAI. En la version auditada, el modo streaming conserva como `raw` el ultimo chunk con `usage=None`; el modo no streaming devuelve la respuesta completa con el `usage` real. No cambia prompts, numero de llamadas ni contenido, y evita estimaciones de tokens.
- Auditoria de ARCH_01 a ARCH_05: las 25 implementaciones desembocan en los mismos adaptadores instrumentados que ARCH_06 y ARCH_07. LangGraph y el cliente especial de handoffs de Microsoft usan `OpenAIInstrumentedLLM`; CrewAI envuelve ese mismo cliente en su `BaseLLM`; Microsoft ARCH_01-04, LlamaIndex y Pydantic AI normalizan el `usage` de sus resultados nativos. En ejecuciones OpenAI todas las llamadas deben declarar `token_counting_method=openai_usage`; el smoke de ARCH_01 y los tests estructurales fallan si una arquitectura evita estos entrypoints.
- Los JSON OpenAI creados antes de esta normalizacion que declaren `token_counting_method=whitespace_proxy` son resultados historicos no comparables y no se convierten retroactivamente. Deben excluirse por metadata o regenerarse con el codigo actual antes del analisis estadistico.
- `ARCH_08_DEBATE_JUDGE` usa tres propuestas independientes sobre el mismo caso, una unica ronda explicita de critica cruzada y un unico juez final. No introduce supervisor, routing, handoffs, particionado documental ni bucle de reflexion/correccion.
- Las propuestas iniciales se ejecutan concurrentemente mediante primitivas nativas porque no comparten estado ni resultados. `debate_round` actua como primera barrera causal y `judge` como segunda; el orden normalizado de trazas sigue siendo `debater_a`, `debater_b`, `debater_c`, `debate_round`, `judge`.
- La ronda de debate se concentra en una sola llamada LLM que revisa cruzadamente las tres propuestas y produce una critica por debater, consenso y desacuerdos. Se evita una llamada de replica por par para fijar cinco llamadas comparables por ejecucion y eliminar sesgos de orden; no se considera fan-in simple porque el paso genera desacuerdos y evaluacion explicita antes del juez.
- No se modifica ningun schema comun para ARCH_08. `DebateProposal`, `CrossCritique`, `DebateRoundOutput` y `JudgeDecision` son modelos internos validados; las metricas aditivas se guardan en `structured_output.debate_execution` y se copian a `ExperimentMetrics.metadata.debate_execution`.
- LangGraph usa `StateGraph` con fan-out desde `START`, arista de origen multiple hacia `debate_round` y arista posterior a `judge`. Microsoft usa `WorkflowBuilder.add_fan_out_edges`, `add_fan_in_edges` y un edge secuencial al juez. LlamaIndex usa eventos de `Workflow`, `Context.collect_events`, debate y juicio separados. Pydantic AI usa `GraphBuilder`, fork, `Join` y pasos tipados para propuesta, debate y decision.
- CrewAI ARCH_08 usa tres tareas asincronas independientes, una tarea `debate_round` con contexto explicito y una tarea `judge` con propuestas y debate como contexto. Usa `Process.sequential`, `allow_delegation=False`, memoria/cache/planning/checkpoint desactivados y ningun `manager_agent`. Las descripciones contienen referencias de contexto porque CrewAI materializa las tareas antes de `kickoff`; los outputs reales se inyectan mediante `Task.context` y se conservan completos en las trazas normalizadas.
- El smoke OpenAI sintetico de ARCH_08 ejecuta cinco llamadas por framework (25 en total) sobre un caso ambiguo y valida propuestas independientes, barreras temporales, critica explicita, decision exclusiva del juez, tokens reales `openai_usage` y persistencia raw. La ejecucion del 22 de julio de 2026 finalizo correctamente en los cinco frameworks; las decisiones variaron entre `select` y `combine`, como corresponde a una arquitectura no determinista sobre un caso con varias soluciones razonables.
- `ARCH_09_REFLECTION_CRITIC_LOOP` mejora una unica respuesta mediante el ciclo secuencial `generator -> critic -> stop_controller -> reviser -> critic`. No crea propuestas independientes ni usa juez, supervisor, routing, handoffs, Map-Reduce o fan-out.
- El limite especifico se configura con `config.metadata["reflection_max_iterations"]`, acepta el alias `max_iterations`, usa `2` por defecto y queda siempre acotado por `config.max_agent_iterations`. El umbral `reflection_quality_threshold` vale `0.85` por defecto. No se modifica `ExperimentConfig` ni ningun schema comun.
- El `stop_controller` es logica determinista y no consume una llamada LLM. Para mantener un motivo estable, prioriza ausencia de defectos criticos, score suficiente y severidad menor; despues aplica el maximo de iteraciones. Cada decision queda registrada como `AgentStep` y en `structured_output.stop_decisions`.
- `benchmark_core.reflection_critic_loop` contiene solo configuracion, prompts, modelos intermedios validados, parsing, politica pura de parada, baseline determinista, trazas y metricas. La orquestacion y el estado operativo permanecen en cada framework. Las metricas aditivas se guardan en `structured_output.reflection_execution` y se copian a `ExperimentMetrics.metadata.reflection_execution`.
- LangGraph ARCH_09 usa un `StateGraph` ciclico con conditional edge desde `stop_controller`; Microsoft Agent Framework usa executors y `WorkflowBuilder.add_switch_case_edge_group`; LlamaIndex usa `Workflow`, eventos y `StopEvent`; Pydantic AI usa `GraphBuilder`, `Step` y `Decision` tipados.
- CrewAI ARCH_09 usa un bucle `for` externo minimo y acotado porque la API de `Crew` no ofrece un ciclo dinamico limpio sin activar manager, planning o una abstraccion de supervisor. Cada generator, critic o reviser sigue siendo un `Agent` y una `Task` nativos ejecutados en una Crew secuencial de una sola tarea, con `max_iter=1`, delegacion, memoria, cache y planning desactivados. Esta es la unica desviacion de representacion del ciclo y no cambia prompts, llamadas ni trazas comparables.
- El baseline local de ARCH_09 usa dos iteraciones: generator, critic inicial, stop, una revision, segundo critic y stop por calidad. Un test adicional fija el maximo en una iteracion y valida la parada por limite sin reviser. No se habilitan bucles ocultos, memoria persistente ni reintentos no trazados.
- La auditoria de warnings del 22 de julio de 2026 confirma que los `DeprecationWarning` de CrewAI 1.14.6 proceden de accesos internos de la propia libreria a campos marcados como deprecated (`function_calling_llm`, `allow_code_execution`, `reasoning` y `multimodal`). El benchmark no configura esas capacidades. No se silencian globalmente ni se actualiza CrewAI durante esta entrega para no cambiar la version experimental sin una regresion separada.
- Los dos `ExperimentalWarning` de Microsoft Agent Framework 1.9.0 se emiten al importar su API publica, que reexporta `SkillResource` y `MemoryStore`. ARCH_09 no usa skills ni memoria/harness; por tanto son avisos de estabilidad de superficies ajenas al flujo ejecutado, no errores del workflow. Se conservan visibles y se recomienda reevaluarlos al actualizar el conjunto de paquetes Microsoft de forma coordinada.
- Pydantic AI si emitia un aviso accionable: el alias `openai:` cambiara de Chat Completions a Responses en v2. Se sustituye por `openai-chat:` para fijar la API utilizada y mantener comparabilidad. El smoke real posterior confirma tokens `openai_usage` sin ese aviso.
- El smoke OpenAI sintetico de ARCH_09 del 22 de julio de 2026 finaliza correctamente en los cinco frameworks. Todos paran por `quality_sufficient` tras la primera critica (dos llamadas y ninguna revision), con scores de critic entre 0.95 y 0.96. Esta ejecucion valida la parada temprana real; la rama con reviser y la parada por maximo siguen cubiertas por los tests deterministas.
- `ARCH_10_CHECKPOINT_MEMORY_RECOVERY` fija una unica frontera logica despues de `planning_or_analysis_step`, inyecta un `ControlledFailure` determinista y ejecuta la continuacion solo desde estado recuperado y validado. La arquitectura realiza exactamente dos llamadas LLM comparables: analisis previo y sintesis posterior.
- No se modifican los schemas comunes. El estado interno `RecoveryWorkflowState`, el fallback `PortableCheckpoint` y los outputs LLM intermedios son modelos Pydantic propios de ARCH_10. El resumen aditivo se guarda en `structured_output.recovery_execution` y se copia a `ExperimentMetrics.metadata.recovery_execution`.
- El error controlado se registra en `failure_injector.error` y en `controlled_error_count`; no se incorpora a `ExperimentResult.errors` porque ha sido recuperado y el resultado nominal es `success`. Cualquier error de persistencia, integridad o continuacion no capturado sigue produciendo el `ExperimentResult` de error habitual del wrapper.
- Todos los checkpoints verifican la presencia del analisis, el stage y un digest SHA-256 del estado canonico. CrewAI y Pydantic escriben de forma atomica un JSON local bajo `results/checkpoints/{framework}/ARCH_10_CHECKPOINT_MEMORY_RECOVERY/`. LlamaIndex guarda en esa ruta el snapshot JSON producido por su `Context`.
- LangGraph ARCH_10 usa `StateGraph` con `SqliteSaver` (`langgraph-checkpoint-sqlite==3.1.0`). Tras el fallo se captura el snapshot, se cierra la conexion SQLite, se abre una conexion nueva, se recompila el grafo y se reanuda mediante la configuracion persistida. Esto demuestra durabilidad local fuera de la memoria Python. SQLite se considera adecuado para el experimento y workflows locales, no un backend de produccion concurrente; una validacion productiva separada deberia usar PostgreSQL. Se autoriza explicitamente la deserializacion de los tres modelos benchmark usados en el estado para mantener una lista msgpack estricta.
- Microsoft Agent Framework ARCH_10 usa `FileCheckpointStorage` y `workflow.run(checkpoint_id=...)`. El framework crea checkpoints nativos al final de cada superstep; `checkpoint_writer` identifica la unica frontera metodologica y `native_checkpoints_created` separa los snapshots tecnicos del conteo comun `checkpoints_created=1`.
- LlamaIndex ARCH_10 usa `Workflow` y la serializacion nativa `Context.to_dict()/from_dict()`. Se divide en una fase pre-checkpoint y otra post-recovery: el estado se recupera del store del contexto restaurado y se entrega a la segunda fase. Esta desviacion evita reinyectar el mismo fallo desde una cola pendiente y mantiene siete pasos y dos llamadas.
- CrewAI 1.14.6 incorpora checkpointing generico de runtime, pero este serializa la entidad completa y se activa mediante eventos. ARCH_10 conserva `Agent`, `Task` y `Crew` nativos para las dos fases LLM, y usa un fallback JSON explicito alrededor de ellas para no persistir agentes/clientes ni introducir disparadores automaticos diferentes. Delegacion, memoria, cache, planning, manager y checkpoint automatico de Crew permanecen desactivados.
- Pydantic AI + pydantic-graph usa dos `GraphBuilder` tipados, uno antes del fallo y otro desde `recovery_loader`. Como `pydantic-graph` no aporta un backend durable comparable, el estado y el checkpoint se validan con Pydantic y se persisten mediante el mismo fallback JSON controlado que CrewAI. La ejecucion durable de Pydantic AI se obtiene integrando Temporal, DBOS, Prefect u otro motor externo; no se incorpora a ARCH_10 porque introduciria infraestructura, semantica de reintentos y dependencias exclusivas de un framework, rompiendo la comparabilidad y la restriccion de no depender de servicios externos.
- Se anade `scripts/run_arch10_openai_smoke.py`: ejecuta el mismo caso con OpenAI en los cinco frameworks, valida las siete trazas, dos llamadas, tokens `openai_usage`, checkpoint, fallo, recuperacion, digest y persistencia raw, e imprime una tabla comparable.
- No se usan memoria conversacional, servicios cloud, LangSmith, CrewAI Cloud, debate, reflexion, supervisor, routing, handoffs, Map-Reduce ni fan-out en ARCH_10.
